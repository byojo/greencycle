# 绿循环 · 技术方案详细文档

> 版本：v1.0  ·  日期：2026-06-28

## 一、产品概述

### 1.1 定位

绿循环是一款基于微信小程序的旧物回收应用，聚焦"上门当面评估"的轻交易模式：

- **C 端用户**：拍照下单 → 骑手上门 → 当面评估 → 当面转账
- **核心理念**：避免线上定价纠纷，提升信任感
- **差异化**：碳积分激励 + 改造故事内容运营

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| 8 大品类 | 手机、衣物、数码、家电、奢侈品、书籍、废品等 |
| 动态表单 | 每个品类的填写字段不同，后端配置 |
| 拍照上传 | 用户上传物品照片，回收员上门评估 |
| 价格透明 | 用户在提交订单前看不到价格，由人工评估 |
| 碳积分体系 | 完成回收获得积分，可兑换/提现 |
| 内容运营 | 改造故事，拉新 + 留存 |

```

1. 小程序选择品类 → 2. 拍照上传（直传 COS）→ 3. 提交订单（POST /orders）
```

## 三、关键技术决策

### 3.1 价格不在前端展示

**决策**：回收总价不在小程序展示，由回收员上门当面评估。

**原因**：
- 避免线上预估价与实际价差距过大引起用户投诉
- 增强信任（用户同意后才交易）
- 灵活定价（回收员可根据实际成色调整）

**实现**：
- 下单页和订单详情页不显示价格字段
- 订单列表仅显示"回收类别 + 时间"
- 订单详情页"回收金额"显示"👨‍🔧 上门当面评估"

### 3.2 品类动态字段

**决策**：每个品类的填写字段不同，由后端动态配置。

**原因**：
- 手机需要品牌型号，衣物需要重量件数，家电需要拆机选项
- 字段增减不需要发版
- 支持后续扩展更多品类

**实现**：
- `category_fields` 表存储字段配置
- 客户端根据 `field_type` 渲染对应组件
- 支持 `text/select/multi/number/switch`

### 3.3 微信登录 + JWT

**决策**：服务端不存储 session，使用 JWT。

**原因**：
- 无状态服务，易于水平扩展
- 移动端友好，避免 CSRF

**实现**：
- 客户端 `wx.login()` 拿 code
- 服务端 `code2Session` 换 openid
- 生成 JWT 返回给客户端
- 客户端每次请求带 `Authorization: Bearer <token>`
- 服务端 401 自动重登

### 3.4 图片直传 COS

**决策**：客户端直传 COS，不经过服务端。

**原因**：
- 减轻服务端带宽压力
- 提高上传速度
- 节省服务端存储

**实现**：
- 客户端请求 `/upload/sign` 拿临时签名
- 用签名直接 PUT 到 COS
- 业务层只存 URL（key）

### 3.5 订单状态机

```
┌──────┐    派单    ┌────────┐   骑手取件  ┌────────┐  当面转账  ┌─────────┐
│ 待评估 ├───────>│ 已派单  ├──────────>│ 已取件  ├─────────>│ 已完成  │

└──────────┘
```

## 四、安全设计

### 4.1 鉴权

- **JWT**：HS256 签名，30 天过期
- **Token 传递**：Header `Authorization: Bearer <token>`
- **Refresh Token**：暂不实现，过期后重新微信登录

### 4.2 防刷

- **Nginx 限流**：每 IP 30 req/s
- **业务限流**：登录、提交订单等敏感接口单独限流（开发中）
- **幂等性**：订单提交带幂等 token（开发中）

### 4.3 数据安全

- **手机号脱敏**：列表显示 `138****6688`
- **隐私彻底清除**：订单完成时提示用户自行清除数据（业务承诺）
- **HTTPS**：强制 HTTPS，HSTS

### 4.4 SQL 注入防护

- 全部使用 GORM 参数化查询
- 禁止字符串拼接 SQL

## 五、性能优化

### 5.1 数据库

| 表 | 索引 |
|----|------|
| users | `open_id`(unique), `phone`, `deleted_at` |
| orders | `user_id+status+created_at`, `category_code`, `rider_id` |
| order_images | `order_id+sort` |
| carbon_point_logs | `user_id+created_at` |

### 5.2 缓存策略

| 数据 | 缓存 |
|------|------|
| 品类列表 | Redis (TTL 1h) |
| 字段配置 | Redis (TTL 1h) |
| 故事列表 | Redis (TTL 30min) |
| 用户信息 | 业务层缓存 (TTL 10min) |

### 5.3 并发

- **数据库连接池**：MaxOpenConns 100
- **Redis 连接池**：PoolSize 100
- **API 实例**：可水平扩展，通过 Nginx upstream 负载均衡
